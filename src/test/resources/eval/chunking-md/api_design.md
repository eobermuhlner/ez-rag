# API Design Principles and Protocols

## Introduction

Application Programming Interface (API) design is the discipline of defining the contract between a service and its consumers. A well-designed API hides implementation complexity, exposes stable abstractions, and evolves without breaking existing clients. Poor API design imposes hidden costs that compound over time: breaking changes require coordinated client migration, unclear semantics cause misuse, and inconsistent error handling forces clients to implement defensive workarounds [Lauret, 2019]. This document surveys six API styles — REST Levels 0 through 3, GraphQL, and gRPC — comparing them across technical and operational dimensions and providing implementation guidance for teams selecting or evolving their API strategy.

## API Protocol Comparison

Choosing an API protocol involves balancing developer experience, operational characteristics, and client compatibility across the full lifecycle of the system. The following table compares six major API styles across eight dimensions. Column definitions: *Typing* refers to whether the protocol enforces typed schemas; *Caching* describes support for standard HTTP caching mechanisms; *Overfetching* indicates whether clients receive data they did not request; *Discoverability* refers to whether clients can navigate the API without out-of-band documentation; *Transport* identifies the underlying wire protocol; *Tooling Maturity* indicates the breadth of client generation, testing, and observability tools available in the open-source ecosystem; *Breaking Change Detection* describes whether schema evolution tooling can flag incompatible changes before they reach production clients.

| API Style           | Typing              | Caching                               | Overfetching                          | Discoverability                               | Transport                                    | Tooling Maturity | Breaking Change Detection  |
|---------------------|---------------------|---------------------------------------|---------------------------------------|-----------------------------------------------|----------------------------------------------|------------------|----------------------------|
| REST Level 0 (POX)  | None                | None — single endpoint, POST-only     | High — full response always returned  | None — requires out-of-band documentation     | HTTP/1.1 or HTTP/2                           | Low              | Manual only                |
| REST Level 1        | Informal JSON/XML   | Partial — resource-specific URLs only | High — fixed response shape           | Poor — resource list available, no traversal  | HTTP/1.1 or HTTP/2                           | Medium           | Manual diff required       |
| REST Level 2        | Informal JSON/XML   | Full — ETags, Cache-Control supported | High — fixed response shape           | Poor — OpenAPI spec required externally       | HTTP/1.1 or HTTP/2                           | High             | OpenAPI diff tools         |
| REST Level 3 (HATEOAS) | Informal JSON/XML | Full — standard HTTP caching works   | High — fixed response shape           | Full — clients navigate via hypermedia links  | HTTP/1.1 or HTTP/2                           | Medium           | Hypermedia schema diff     |
| GraphQL             | Strong — SDL schema | None — POST-only, no HTTP caching     | Eliminated — client specifies exact fields | Good — introspection query available      | HTTP/1.1 (typically)                         | High             | Schema breaking change linter |
| gRPC                | Strong — Protobuf   | None — streaming, no HTTP caching     | None — precise field selection        | Poor — reflection optional, not default       | HTTP/2 binary framing with Protocol Buffers  | High             | Protobuf compatibility rules |

The table presents a snapshot of each style's characteristics at their canonical implementation; real-world deployments frequently combine elements from multiple levels. For example, a team might implement Level 2 REST for public APIs while using gRPC for internal service-to-service communication, exploiting gRPC's efficiency advantages within the trusted network boundary while offering a widely understood REST surface to external consumers. The choice of API style also interacts with the team's existing skills, the consumer landscape (browser clients require HTTP semantics that gRPC cannot directly satisfy), and the organization's observability tooling which may have better support for HTTP traffic than for binary Protobuf streams. Organizations that expose APIs to external third-party developers must additionally consider developer experience: REST with OpenAPI documentation has a significantly lower barrier to entry for external developers than gRPC, which requires familiarity with Protobuf toolchains and generates less human-readable wire traffic during debugging.

The REST maturity model, described by Richardson and popularized by Fowler, provides a useful progression from ad hoc HTTP usage toward a fully hypermedia-driven interface. Most production APIs operate at Level 2, which provides meaningful HTTP verb semantics and status codes without the complexity of HATEOAS link generation. Level 2 APIs use GET for idempotent reads, POST for resource creation, PUT for full replacement, PATCH for partial updates, and DELETE for removal; each operation returns a semantically appropriate HTTP status code that clients can interpret without parsing the response body. The discipline of using correct status codes — 201 for creation, 204 for successful delete, 409 for conflict, 422 for validation failure — significantly reduces the client-side code required to handle error conditions, because clients can branch on the status code class (2xx, 4xx, 5xx) before inspecting the response body. Additionally, Level 2 APIs that return consistent problem-detail objects (as specified in RFC 7807) for all error responses allow client code to handle errors uniformly rather than writing format-specific parsing logic for each endpoint's error format.

The ETag header enables conditional GET requests by transmitting a hash of the resource state; a client that supplies If-None-Match with the current ETag value receives HTTP 304 Not Modified when the resource has not changed, reducing bandwidth consumption by up to 80 percent on read-heavy APIs. This mechanism is particularly valuable for mobile clients operating on metered connections and for intermediate caches such as CDN edge nodes that can serve cached responses without forwarding requests to the origin server. Implementing ETags requires the server to compute a fingerprint of the resource (typically a hash of the serialized JSON body or the database row's updated_at timestamp) and include it in every GET response as the ETag header. The Last-Modified header provides a weaker alternative based on timestamps rather than content hashes; ETag is preferred because it correctly handles sub-second updates and cases where content is identical but the timestamp has changed due to a spurious write.

API rate limiting should communicate its policy through the Retry-After and X-RateLimit-Remaining response headers, allowing well-behaved clients to back off gracefully rather than repeatedly retrying after receiving HTTP 429 Too Many Requests. The Retry-After header specifies either a delay in seconds or an absolute HTTP date after which the client may retry; X-RateLimit-Remaining conveys how many requests remain within the current window; X-RateLimit-Reset conveys the Unix timestamp at which the window resets. Clients that respect these headers reduce load on the server during congestion periods by self-throttling, improving overall system stability without requiring server-side queuing of requests. Rate limiting policies should be applied per authenticated identity rather than per IP address, because clients behind corporate NAT gateways may share a single IP address across hundreds of users, making IP-based limiting unfairly restrictive.

GraphQL is most beneficial when multiple client types — web, mobile, and third-party integrations — need to query the same data with different field subsets, because the SDL schema provides a single typed contract from which all clients generate typed query interfaces. The GraphQL execution engine resolves only the fields requested in each query, eliminating the overfetching that plagues Level 1 and Level 2 REST APIs where every call returns the full resource representation regardless of how many fields the client actually uses. The performance characteristic of GraphQL that trips up new adopters is the N+1 query problem: a query that fetches a list of resources and then for each resource fetches a related resource naively issues N+1 database queries. The DataLoader pattern — batching and deduplicating sub-resource fetches across an entire query execution — resolves this at the cost of adding complexity to every resolver that fetches related data. Production GraphQL deployments require query complexity analysis and depth limiting to prevent clients from constructing queries that would trigger millions of database operations.

gRPC is the preferred API style for synchronous service-to-service communication inside a trusted network, because Protobuf binary serialization is approximately six times more compact than equivalent JSON and HTTP/2 multiplexing eliminates the head-of-line blocking that affected HTTP/1.1-based REST when services issue multiple concurrent requests over a single connection. The Protobuf schema language enforces backward and forward compatibility rules: adding a new field with a new field number is always backward compatible because existing consumers ignore unknown fields; removing a field or reusing its field number is always backward incompatible. These rules, enforced by Protobuf compatibility linters integrated into CI pipelines, make breaking change detection in gRPC significantly more reliable than in JSON-based REST APIs, where field removals and type changes can go undetected until a consumer fails at runtime.

API gateways serve as the single entry point for all external API traffic, providing cross-cutting capabilities — authentication and authorization, rate limiting, request transformation, response caching, and telemetry collection — without requiring individual services to implement them independently. The gateway pattern is particularly valuable in microservices deployments where tens or hundreds of services each expose their own endpoints: without a gateway, every external consumer must authenticate separately with every service, and there is no central point at which to enforce consistent security policies across all APIs. Kong, AWS API Gateway, and Azure API Management are widely adopted commercial options; Envoy proxy configured as an edge gateway is a popular open-source alternative. The primary operational risk of an API gateway is that it becomes a single point of failure: every external request passes through the gateway, and if the gateway is unavailable or saturated, all external traffic fails regardless of the health of the underlying services. This risk is mitigated by running the gateway as a horizontally scaled cluster with health checks and automatic failover rather than as a single instance.

OpenAPI (formerly Swagger) has emerged as the de facto standard for describing REST API contracts in a machine-readable format. An OpenAPI specification document — written in JSON or YAML — defines every endpoint's path, HTTP method, request parameters, request body schema, and response schemas. This machine-readable contract enables a wide ecosystem of tooling: client SDK generation in any language, interactive documentation via Swagger UI or Redoc, contract testing via tools such as Dredd or Pact, and API linting via Spectral. The most valuable use of an OpenAPI specification in a development workflow is as a design-first artifact: teams agree on the API contract in the specification before any implementation code is written, allowing frontend and backend teams to work in parallel using generated mocks from the specification. API linting rules — enforcing conventions such as kebab-case path segments, pagination on all list endpoints, and consistent error response schemas — catch design inconsistencies before the API reaches consumers.

Authentication and authorization are the most security-critical aspects of API design. The OAuth 2.0 framework, combined with OpenID Connect for identity, provides a standardized authorization flow for APIs that must act on behalf of a user. Bearer tokens issued by an authorization server are transmitted via the Authorization: Bearer header; the API validates the token signature and expiration on every request without maintaining server-side session state. Token validation should verify the signature using the authorization server's published public keys (retrieved from the JWKS endpoint), the audience claim (to ensure the token was issued for this API rather than another), and the expiration time. APIs that store sensitive data — personal information, financial records, health records — must additionally implement field-level authorization: the token may authorize a user to access a resource, but the response must be filtered to exclude fields that the user's role does not permit them to read. Failing to implement field-level authorization exposes sensitive data to users who have legitimate access to the resource but not to all its attributes.

Pagination is a fundamental requirement for any API endpoint that returns a collection of resources. Without pagination, a collection that grows to millions of records will eventually cause response timeouts, excessive memory consumption on the server, and bandwidth exhaustion on the client. Cursor-based pagination — where each page includes an opaque cursor value that encodes the position of the last item returned, which the client supplies as a query parameter to fetch the next page — is more robust than offset-based pagination for collections where new items are inserted between pages. Offset-based pagination (`?page=3&pageSize=20`) can skip items or return duplicates when the underlying collection changes between page fetches; cursor-based pagination avoids this because the cursor points to a stable position in the sorted order rather than to an absolute offset. REST Level 2 and Level 3 APIs should document their pagination mechanism in their OpenAPI specification and return pagination metadata (total count, next page cursor or URL) in a standard envelope or Link header. The Link header format defined in RFC 5988 allows servers to embed next, prev, first, and last URL references directly in the response headers, enabling clients to navigate paginated collections without parsing the response body for pagination metadata — an approach that is compatible with partial content downloads and streaming consumers that process pages as they arrive rather than loading the full response into memory.

## API Versioning Strategies

API versioning is the mechanism by which a service evolves its interface without immediately breaking existing consumers. Three main strategies are in common use: URI path versioning (`/v1/users`, `/v2/users`), header versioning (`Accept: application/vnd.example+json; version=2`), and query parameter versioning (`/users?version=2`). URI path versioning is the most widely adopted because it is visible in browser address bars, logs, and firewall rules, and because existing clients continue to work without modification until the old version is explicitly decommissioned.

The deprecation lifecycle for API versions should be defined before the API launches. A typical enterprise policy reserves the right to remove a version after it has been deprecated for at least eighteen months and after active usage has fallen below 1% of total API calls. Consumers should receive deprecation notices through both response headers (`Deprecation: true`, `Sunset: Sat, 01 Jan 2028 00:00:00 GMT`) and out-of-band developer communications delivered with at least twelve months of advance notice.
